# Prozessbasierte Softwareentwicklung: „Traditionelle Entwicklung ist am Ende“

Die Aussage „Traditionelle Entwicklung ist am Ende“ (Golem.de) ist bewusst zugespitzt formuliert. Gemeint ist in der Regel nicht, dass Softwareentwicklung verschwindet – sondern dass sich das Paradigma verschiebt.

## 1. Was ist „traditionelle Entwicklung“?

Damit ist meist gemeint:

- Feature → Ticket → Entwickler → Code
- Mensch schreibt nahezu jede Zeile selbst
- Tools unterstützen, aber entscheiden nicht
- Dokumentation entsteht nach oder neben dem Code
- Architektur ist relativ statisch
- AI ist maximal Assistenz (Autocomplete)

Kurz: code-zentrierte Entwicklung.

## 2. Was bedeutet „prozessbasiert“?

Prozessbasiert heißt:

Nicht der Code steht im Zentrum – sondern der Workflow / Prozess, der:

- Anforderungen strukturiert
- Spezifikationen erzeugt
- Code generiert
- Tests erstellt
- Reviews automatisiert
- Qualität prüft
- Änderungen nachvollziehbar macht

Der Mensch definiert Ziele, Regeln, Constraints und Architekturprinzipien. Die KI bzw. automatisierte Systeme führen aus.

Das ist ein Shift von „Ich schreibe Code“ zu „Ich designe Systeme, die Code erzeugen“.

## 3. Was ändert sich konkret?

**Vorher:**  
Idee → Entwickler tippt → PR → Review → Merge

**Prozessbasiert:**  
Idee → Spec → Validierung → Generierung → Tests → Gate → Review

Code ist nur noch ein Artefakt im Prozess.

## 4. Warum sagen manche: „Traditionell ist am Ende“?

Weil AI-Modelle heute:

- vollständige Module erzeugen
- Tests generieren
- Architektur-Vorschläge machen
- Refactorings durchführen
- Code Reviews automatisisieren

Wenn 70–80 % mechanische Arbeit automatisierbar sind, verschiebt sich der Fokus:

| Früher            | Heute                    |
|-------------------|--------------------------|
| Syntax-Können     | Systemdenken             |
| Framework-Details | Architektur-Design       |
| Implementieren    | Orchestrieren            |
| Code schreiben    | Spezifikationen definieren |

## 5. Was steckt strategisch dahinter?

Das ist kein Tool-Wechsel, sondern ein Strukturwandel:

- weg von Human-driven Implementation
- hin zu Process-driven System Construction

Das passt stark zu dem Gedanken: KI nicht als Ergänzung, sondern als strukturbildendes Fundament.

## 6. Ist das wirklich „das Ende“?

Realistisch betrachtet:

- Klassisches Coding verschwindet nicht.
- Aber es wird weniger dominierend.
- Die Rolle des Entwicklers verschiebt sich Richtung:
  - Systemarchitekt
  - Prozessdesigner
  - Qualitäts-Gatekeeper
  - Kontext-Engineer

## 7. Besonders relevant für dich

Wenn man dein Interesse an folgenden Themen betrachtet, bewegst du dich bereits exakt in diese Richtung:

- Spec-Driven Development
- AI-Agenten
- deterministischen Runnern
- Facade-only-Architektur
- automatisierten Gates

Du denkst schon prozessbasiert.

## 8. Der eigentliche Kern der Aussage

„Traditionelle Entwicklung ist am Ende“ heißt eher:

Software wird nicht mehr primär durch manuelles Codieren erschaffen, sondern durch die Gestaltung intelligenter, automatisierter Entwicklungsprozesse.
